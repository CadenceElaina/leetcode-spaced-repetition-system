## 1. Overview

LeetRepeat is a free, open-source web application that combines structured LeetCode attempt logging with spaced repetition scheduling and interview readiness scoring. It is built around the NeetCode 150 problem set and uses a modified FSRS algorithm to schedule reviews based on demonstrated retention — not just completion.

### The Gap

| Tool           | Tracks Completion | Tracks Retention | Spaced Repetition | Code + Notes | Complexity Validation | Readiness Score |
| -------------- | :---------------: | :--------------: | :---------------: | :----------: | :-------------------: | :-------------: |
| LeetCode       |         ✓         |        ✗         |         ✗         |   Partial    |           ✗           |        ✗        |
| NeetCode       |         ✓         |        ✗         |         ✗         |      ✗       |           ✗           |        ✗        |

## 2. Goals & Non-Goals

### Goals

- Provide a free tool that any CS student or career changer can use to structure their interview prep
- Use a science-backed algorithm to schedule reviews so users practice what they're forgetting, not what they already know
- Give users an honest, data-driven answer to "am I ready for interviews?"
- Store structured attempt data (code, notes, complexity answers, confidence) to support meaningful self-review
- Support the NeetCode 150 problem set at launch, extensible to NeetCode 250 and custom lists

### Non-Goals

- **Not a LeetCode clone.** We do not host problem statements, test cases, or execute code. Users solve problems on LeetCode and log results here.
- **Not a teaching tool.** We do not explain solutions or algorithms. NeetCode, editorial solutions, and YouTube already do that well. We track whether you've retained what you learned.
- **Not a social platform.** No leaderboards, no competitive features. This is a personal tracking tool. Community features (shared notes, discussions) are a future possibility, not an MVP concern.
- **Not a mobile app.** Web-first, mobile-responsive. Native apps are out of scope.

## 3. Target Users

### Primary

- **CS students** (traditional or non-traditional) preparing for internship or new grad interviews
- **Career changers** entering software engineering who need structured interview prep

### Secondary

- **Experienced engineers** returning to the job market after a gap
- **Anyone doing LeetCode** who wants retention-aware practice scheduling

### User Assumptions

- Users are already doing LeetCode or NeetCode problems — they don't need to be convinced to practice
- Users have access to a web browser on desktop (mobile is secondary)
- Users are honest in self-reporting (the system is a personal tool, not a proctored exam)
- Technical literacy: comfortable with a web app, code editors, and LeetCode's interface

## 4. Functional Requirements

### FR-1: Problem Database

- **FR-1.1:** Store the NeetCode 150 problem set with: title, LeetCode number, LeetCode URL, NeetCode URL, NeetCode video ID, category, difficulty, list source, Blind 75 flag, optimal time complexity, optimal space complexity, and pattern tags.
- **FR-1.2:** Seed the database from `problems.json` (generated by the data pipeline).
- **FR-1.3:** Support filtering and sorting by category, difficulty, completion status, retention level, and Blind 75 membership.
- **FR-1.4:** Extensible to NeetCode 250 and user-added custom problems in future.

### FR-2: User Accounts

- **FR-2.1:** Users authenticate via GitHub or Google OAuth (NextAuth.js / Auth.js v5).
- **FR-2.2:** Each user has a private workspace — no user can see another's data.
- **FR-2.3:** Users can set a target interview date.
- **FR-2.4:** Users can export their data (attempts, notes, code) as JSON.

### FR-3: Attempt Logging

- **FR-3.1:** Users log an attempt against a specific problem with the following fields:

| Field                            | Type                                                   | Required | Description                                                              |
| -------------------------------- | ------------------------------------------------------ | :------: | ------------------------------------------------------------------------ |
| Solved independently             | Enum: Yes / Partial / No                               |    ✓     | Did you reach a working solution without external help?                  |
| Solution quality                 | Enum: Optimal / Suboptimal / Brute Force / No Solution |    ✓     | Self-assessed quality of the approach                                    |
| Time complexity (user's answer)  | String (e.g., "O(n log n)")                            |    ✓     | What the user believes their solution's time complexity is               |
| Space complexity (user's answer) | String                                                 |    ✓     | Same for space                                                           |
| Solve time                       | Integer (minutes)                                      | Optional | Time spent working on the problem                                        |
| Study time                       | Integer (minutes)                                      | Optional | Time spent reviewing the solution afterward                              |
| Rewrote from scratch             | Enum: Yes / No / Did Not Attempt                       | Optional | After stepping away, could you rewrite the solution?                     |
| Confidence                       | Integer 1–5                                            |    ✓     | Gut feeling on retention. 1 = "no clue", 5 = "could do this in my sleep" |
| Code                             | Text (multi-line)                                      | Optional | The user's solution, stored as text. Syntax-highlighted in the UI.       |
| Notes                            | Text (Markdown)                                        | Optional | Approach, edge cases spotted, patterns recognized, mistakes made         |

- **FR-3.2:** After submission, the system auto-compares the user's time and space complexity answers against the known optimal values and flags mismatches.
- **FR-3.3:** Attempt history is preserved — users can view all past attempts for any problem.
- **FR-3.4:** The most recent attempt drives the spaced repetition state update.

### FR-4: Spaced Repetition Scheduling

- **FR-4.1:** Each user-problem pair maintains a stability value (S) and a computed retrievability value (R).
- **FR-4.2:** After each attempt, stability is recalculated based on attempt quality signals (see §6).
- **FR-4.3:** The system generates a daily review queue sorted by priority (see §6.3).
- **FR-4.4:** Users can override scheduling — work on any problem at any time, not only those in the queue.

### FR-5: Interview Readiness Scoring

- **FR-5.1:** Compute an overall readiness score from coverage, retention, consistency, and pace metrics (see §7).
- **FR-5.2:** Assign a tier (S/A/B/C/D) based on configurable thresholds.
- **FR-5.3:** Surface weak categories — categories with the lowest average retrievability.
- **FR-5.4:** If a target date is set, compute whether the user's current pace will get them ready in time.

### FR-6: Dashboard

- **FR-6.1:** Display: readiness tier, readiness score, days until target, problems due today, problems due this week.
- **FR-6.2:** Category coverage heatmap showing per-category retention.
- **FR-6.3:** Streak / consistency indicator.
- **FR-6.4:** Quick link to start today's review queue.

### FR-7: Analytics

- **FR-7.1:** Problems solved over time (line chart).
- **FR-7.2:** Retrievability distribution across all attempted problems (histogram).
- **FR-7.3:** Category breakdown with average retention per category.
- **FR-7.4:** Time spent (solve time + study time) trends.
- **FR-7.5:** Improvement trends — how solution quality and confidence change on repeated attempts.

## 6. Spaced Repetition Algorithm

### 6.1 Core Model

Each user-problem pair has a **stability** value S (in days) representing how long the memory will last. **Retrievability** R is the probability the user can still solve the problem today:

$$R(t) = e^{-t / S}$$

Where t = days since last review.

### 6.2 Stability Update

After each attempt, a new stability value is computed:

```
S_new = S_old * multiplier * modifier
```

**Multiplier** (based on attempt outcome):

| Outcome                                                        |   Multiplier    |
| -------------------------------------------------------------- | :-------------: |
| Solved independently = Yes, Quality = Optimal                  |       2.5       |
| Solved independently = Yes, Quality = Suboptimal               |       2.0       |
| Solved independently = Partial, Quality = any working solution |       1.5       |
| Solved independently = No, but studied solution                | 1.0 (no growth) |
| No working solution at all                                     |   0.5 (decay)   |

**Modifier** (additive bonuses applied to the multiplier):

| Signal                            | Modifier |
| --------------------------------- | :------: |
| Rewrote from scratch = Yes        |   +0.5   |
| Time complexity correct           |   +0.2   |
| Space complexity correct          |   +0.2   |
| Confidence 4–5                    |   +0.1   |
| Solve time < 10 min (for mediums) |   +0.2   |
| Confidence 1–2                    |   −0.1   |

**Bounds:** S is clamped to [0.5 days, 365 days]. Initial S for a new problem = 0.5 days.

### 6.3 Review Queue Priority

The daily review queue is sorted by:

```
priority = (1 - R) * weight
```

Where weight combines:

- **List importance:** NeetCode 150 base weight 1.0; Blind 75 subset gets +0.2
- **Category weakness:** If the user's average R in a category is below 0.6, all problems in that category get +0.3
- **Difficulty:** Easy = 0.8, Medium = 1.0, Hard = 1.1

Problems with R < 0.5 are always shown at the top ("overdue"). Problems with R > 0.9 are hidden from the queue unless specifically browsed.

### 6.4 Retrievability Floor

No problem's R drops below 0.3 from decay alone. This prevents long-ago mastered problems from appearing as if the user has never seen them. They'll get a gentle "maintenance review" suggestion but won't pollute the urgent queue.

## 7. Interview Readiness Engine

### 7.1 Metrics

| Metric           | Definition                                                            | Weight |
| ---------------- | --------------------------------------------------------------------- | :----: |
| Coverage         | % of NeetCode 150 problems attempted at least once                    |  25%   |
| Retention        | % of attempted problems with R > 0.7                                  |  35%   |
| Category Balance | Lowest category average R (penalizes having blind spots)              |  20%   |
| Consistency      | % of scheduled reviews completed in the last 14 days                  |  10%   |
| Pace             | Projected date to reach current tier's next threshold at current rate |  10%   |

The readiness score is a weighted sum, scaled to 0–100.

### 7.2 Tier Thresholds

| Tier  | Score Range | Typical Profile                                                          |
| ----- | :---------: | ------------------------------------------------------------------------ |
| **S** |   90–100    | 150 problems, 85%+ retention, all categories strong, consistent schedule |
| **A** |    75–89    | 150 problems, 80%+ retention, no weak categories below 0.6 avg R         |
| **B** |    55–74    | 120+ problems, 70%+ retention, most categories covered                   |
| **C** |    35–54    | 75+ problems, 60%+ retention, working through mediums                    |
| **D** |    0–34     | < 75 problems or low retention                                           |

### 7.3 Target Date Projection

If a target date is set:

- Compute the delta between current score and the next tier threshold
- Estimate required problems/day and reviews/day based on historical pace
- Show: "At your current pace, you'll reach Tier B by [date]" or "You need to increase to X problems/week to hit Tier B by [target date]"

## 10. Data Pipeline

### 10.1 Overview

A Python script (`scripts/fetch_problems.py`) generates `problems.json` from the [neetcode-gh/leetcode](https://github.com/neetcode-gh/leetcode) GitHub repository (MIT licensed). This is a one-time data collection step, re-run only when the upstream data changes.

### 10.2 Process

1. Download `.problemSiteData.json` from GitHub (problem metadata for all 450 NeetCode problems)
2. Filter to `neetcode150 == true` (150 problems)
3. Download the full repository as a ZIP archive (~6 MB, single HTTP request)
4. Extract `hints/*.md` files from the ZIP (150 files, one per NeetCode problem)
5. Map hint filenames to problem slugs using an explicit 74-entry mapping dictionary (NeetCode uses different slugs than LeetCode for 74 of 150 problems)
6. Parse optimal time and space complexity from the HTML hint content via regex
7. Run verification checks: 150 total, 75 Blind 75, 28 Easy / 101 Medium / 21 Hard, 18 categories
8. Output `problems.json`

### 10.3 Current Status

- **150/150 problems** with full metadata
- **150/150 optimal complexity values** extracted and verified
- Dependencies: Python standard library only (no pip packages)

### 10.4 Database Seeding

A Drizzle seed script (to be written) will read `problems.json` and insert/upsert rows into the Problem table. This runs once during initial setup and can be re-run to update problem data.

## 11. Assumptions

| #   | Assumption                                                                        | Risk if Wrong                                        | Mitigation                                                                                                                                                               |
| --- | --------------------------------------------------------------------------------- | ---------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| A1  | Users solve problems on LeetCode and self-report results honestly                 | Inflated readiness scores                            | The tool is personal — cheating only hurts yourself. Notes + code provide a self-check.                                                                                  |
| A2  | NeetCode 150 is a sufficient base for interview prep at tier B and below          | Users may need more problems for S/A tiers           | NeetCode 250 support is planned for Phase 3. Custom problem support is in the roadmap.                                                                                   |
| A3  | Optimal time/space complexity has one canonical answer per problem                | Some problems have multiple valid optimal approaches | Store the most commonly accepted optimal. Add a notes field on Problem for alternate approaches. Flag when the user's answer is valid but different from stored optimal. |
| A4  | Users will self-assess solution quality accurately                                | Miscalibrated stability values                       | The "rewrote from scratch" and "complexity correctness" signals provide objective cross-checks against the subjective quality rating.                                    |
| A5  | The FSRS-based algorithm with our domain multipliers will produce good scheduling | Poor scheduling erodes trust                         | Start with conservative parameters. Log all algorithm inputs/outputs for future tuning. Expose a "this came back too early / too late" feedback button.                  |
| A6  | Supabase free tier is sufficient for individual users                             | DB limits hit                                        | Free tier supports 500 MB storage and 2 GB bandwidth. A single user with 150 problems and thousands of attempts stays well within this.                                  |

## 12. Edge Cases & Mitigations

### Problem Data

| Edge Case                                                   | Handling                                                                                                        |
| ----------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------- |
| NeetCode updates their problem list (adds/removes problems) | Re-run the pipeline. Upsert on `leetcodeNumber`. Removed problems remain in DB but are marked inactive.         |
| A problem has multiple valid optimal complexities           | Store the most common. Add `alternateComplexities` field. Complexity comparison should accept any listed value. |
| Hint file has no extractable complexity (malformed HTML)    | Pipeline reports it. Manual override in `problems.json` as a fallback.                                          |
| LeetCode URL changes or problem is removed                  | Problems remain in our DB with a stale URL. Users can report broken links.                                      |

### User Behavior

| Edge Case                                                        | Handling                                                                                                                                                                                  |
| ---------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| User logs an attempt months after their last one                 | R has decayed to the floor (0.3). Stability reset is appropriate — treat it as near-fresh.                                                                                                |
| User logs multiple attempts for the same problem on the same day | Each attempt is stored. Only the last attempt of the day drives the stability update.                                                                                                     |
| User never logs a "rewrote from scratch"                         | The field is optional. Algorithm uses other signals. Stability growth is slightly more conservative without it.                                                                           |
| User reports "Optimal" but enters wrong complexity               | `timeComplexityCorrect` / `spaceComplexityCorrect` will be false. The stability modifier for complexity correctness won't be applied, partially canceling out the inflated quality claim. |
| User sets a target date in the past                              | Validate on the frontend. Reject dates that are not in the future.                                                                                                                        |
| User changes their target date                                   | Recalculate pace projection immediately. Historical data is unaffected.                                                                                                                   |

### Algorithm

| Edge Case                                             | Handling                                                                                                                                      |
| ----------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------- |
| Stability exceeds 365 days (problem "too mastered")   | Clamped to 365. Still appears in maintenance reviews at very low priority.                                                                    |
| Stability drops below 0.5 days                        | Clamped to 0.5. Problem appears in the next day's queue.                                                                                      |
| User has 0 completed problems (brand new account)     | Dashboard shows tier D with onboarding guidance: "Start with Arrays & Hashing — Easy problems."                                               |
| All problems in a category are overdue simultaneously | Queue caps at a configurable daily limit (default: 20 reviews). Highest priority problems surface first. Remaining roll over to the next day. |

## 14. Security Considerations

| Concern                   | Mitigation                                                                                                                                                       |
| ------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Authentication**        | OAuth only (no password storage). NextAuth handles tokens, CSRF, and session management.                                                                         |
| **Authorization**         | All data queries are scoped to the authenticated user's ID. No user can access another's attempts or state. Server-side enforcement via middleware.              |
| **SQL injection**         | Drizzle ORM uses parameterized queries. No raw SQL string concatenation.                                                                                         |
| **XSS**                   | React's default escaping. User-submitted code is stored as text and rendered in a sandboxed code viewer, never executed. Markdown is sanitized before rendering. |
| **Data privacy**          | Users own their data. Export as JSON is a core feature. No data is shared or sold.                                                                               |
| **Rate limiting**         | API routes should be rate-limited to prevent abuse (Next.js middleware + in-memory store or Upstash Redis in production).                                        |
| **Environment variables** | Database URL, OAuth secrets, and API keys stored in environment variables. Never committed to source. `.env.example` provided with placeholder values.           |

## 15. Performance Considerations

| Area                   | Approach                                                                                                                                                                                |
| ---------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Dashboard load**     | Readiness score and tier computed server-side (server component). Cached per-user per-day with revalidation on new attempt.                                                             |
| **Review queue**       | Single SQL query: `SELECT ... FROM user_problem_state WHERE userId = ? AND nextReviewAt <= NOW() ORDER BY priority DESC LIMIT 20`. Indexed on `(userId, nextReviewAt)`.                 |
| **Problem list**       | Server-side rendering with pagination. 150 problems is small enough to load in one page, but filtersgand sorts should be server-side for responsiveness.                                |
| **Attempt submission** | Single transaction: INSERT Attempt + UPDATE UserProblemState. Sub-100ms at this scale.                                                                                                  |
| **Code storage**       | Plain text in the database. No blob storage needed — individual solutions are small (< 10 KB).                                                                                          |
| **Database scale**     | A single user with 150 problems × 50 attempts each = 7,500 Attempt rows. Thousands of users would still be under 100 MB. Supabase free tier (500 MB) is sufficient for early user base. |
